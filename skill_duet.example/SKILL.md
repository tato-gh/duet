# このスキルの目的

起動中の duet エントリへプロンプトを送信し、応答を受け取る。
各エントリは独立したスレッドコンテキストを持つため、同じエントリに連続送信すると会話の文脈を継続できる。
このスキルは、単発で処理を投げるよりも「同じ文脈で考え続ける伴走相手」として使うと価値が出やすい。

メタファ: 「右目と左目があって立体視というボーナスが得られる」 伴走によるボーナスを得たい。

## エントリの概念

- エントリ1つ = 1つの独立した対話セッション
- 利用可能なエントリ名は `entries.exs` で確認する
- エントリはファイル操作などのツールを持ち、git diff で現在の作業状態も参照できる
- 文脈をリセットしたい場合は `/clear` を送信する
- 文脈を要約して新スレッドに引き継ぎたい場合は `/compact` を送信する

## 基本フロー

このスキルでは、だいたい次の流れで進める。

### 1. 利用可能なエントリ確認

`entries.exs` を使う。

```bash
elixir --sname NAME@localhost /path/to/skill_duet/entries.exs
```

### 2. エントリを選ぶ

- 既存文脈を引き継ぐなら同じエントリを再利用する
- 用件が変わる場合は、送信前に `/clear` を使うか、別エントリへ切り替える

### 3. 単一エントリへ送る

`post.exs` を使う。

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post.exs ENTRY_NAME "PROMPT"
```

長文・複数行の場合は、ヒアドキュメントでプロンプトを送る：

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post.exs ENTRY_NAME "$(cat <<'EOF'
複数行のプロンプト
行2
行3
EOF
)"
```

### 4. 複数エントリへ並行送信する

`post_parallel.exs` を使う。エントリ名とプロンプトを交互に並べる。

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post_parallel.exs ENTRY1 "PROMPT1" ENTRY2 "PROMPT2"
```

長文・複数行の場合は、ヒアドキュメントでプロンプトを送る：

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post_parallel.exs \
  entry1 "$(cat <<'EOF'
プロンプト1
複数行
EOF
)" \
  entry2 "$(cat <<'EOF'
プロンプト2
複数行
EOF
)"
```

## 一般プラクティス

- まずは短く渡す（目的と完了条件）
- 同じPJ前提の説明は省き、差分だけ共有する
- 新しい話題になったら `/clear` して再開する
- 話題の区切りに `/compact` で要約して継続する
- 同じエントリへは「前回からの差分要求」だけ送る
- `entries.exs` で役割が複数見える場合は、名前/roleから適切な運用を選ぶ

## 典型ユースケース（詳しめ）

### 1. 伴走者として使う

自身が進める作業を、同じ文脈の相手に継続相談しながら補強する。

- 最初は短く渡す（目的と完了条件）
- 同じPJ前提の説明は省き、差分だけ共有する
- 複数案が出たら、確信度を `1-5` で添えて会話する
  - `1`: 仮説
  - `3`: 妥当
  - `5`: 強い確信
- 意見が割れたら、案 -> 反証 -> 統合（弁証法）でよりよい案を探る

### 2. 判定者として使う

意見が乱立して決めきれない争点を、論理整合性で判定してもらう。

- 判定依頼は、案が2つ以上あるときに使う
- 判定前に、可能なら弁証法（案 -> 反証 -> 統合）を一度通す
- 判定依頼時は、短く材料を渡す
  - 論点
  - 各案
  - 根拠
  - 制約
  - 残る懸念
- 伴走時の確信度（`1-5`）は、判定依頼には持ち込まない

## 継続対話の例

- しりとりのような継続対話も、同じ相手を使い続けると安定する

## 依頼の例

伴走:

```text
目的: ここから実装方針を詰めたい
完了条件: 採用案と懸念が整理されている
```

判定:

```text
論点: キャッシュ戦略を A/B どちらにするか
案A: 実装が単純、運用負荷が低い
案B: 性能は高いが複雑
根拠: 現在の運用体制は夜間対応1名
制約: 切り戻し手順が5分以内であること
残る懸念: 障害時の切り戻し
論理整合性で最終判断してほしい
```

## エラー

| reason | 意味 | 対処 |
|--------|------|------|
| `:not_found` | エントリ名が存在しない | `entries.exs` で利用可能なエントリ名を再確認する |
| `:busy` | エントリが処理中 | 待ってリトライする |
| `"failed"` / `"interrupted"` | LLM 側でエラー | リトライする |
