# このスキルの目的

起動中の duet エントリへプロンプトを送信し、応答を受け取る。
各エントリは独立したスレッドコンテキストを持つため、同じエントリに連続送信すると会話の文脈を継続できる。
このスキルは、単発で処理を投げるよりも「同じ文脈で考え続ける伴走相手」として使うと価値が出やすい。

## エントリの概念

- エントリ1つ = 1つの独立した対話セッション
- 利用可能なエントリ名は `entries.exs` で確認する
- 文脈をリセットしたい場合は `/clear` を送信する

## 基本フロー

このスキルでは、だいたい次の流れで進める。

### 1. 利用可能なエントリ確認

`entries.exs` を使う。

```bash
elixir --sname NAME@localhost /path/to/skill_duet/entries.exs
```

### 2. エントリを選ぶ

- 既存文脈を引き継ぐなら同じエントリを再利用する
- 用件が変わる場合は、送信前に `/clear` を使うか、別エントリへ切り替える

### 3. 単一エントリへ送る

`post.exs` を使う。

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post.exs ENTRY_NAME "PROMPT"
```

長文・複数行の場合は、ヒアドキュメントでプロンプトを送る：

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post.exs ENTRY_NAME "$(cat <<'EOF'
複数行のプロンプト
行2
行3
EOF
)"
```

### 4. 複数エントリへ並行送信する

`post_parallel.exs` を使う。エントリ名とプロンプトを交互に並べる。

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post_parallel.exs ENTRY1 "PROMPT1" ENTRY2 "PROMPT2"
```

長文・複数行の場合は、ヒアドキュメントでプロンプトを送る：

```bash
elixir --sname NAME@localhost /path/to/skill_duet/post_parallel.exs \
  entry1 "$(cat <<'EOF'
プロンプト1
複数行
EOF
)" \
  entry2 "$(cat <<'EOF'
プロンプト2
複数行
EOF
)"
```

## 協働のスタンス

- まずは「最小限の共有」で任せる（目的と完了条件があれば十分なことが多い）
- 同じPJを見ている前提なので、重複説明は省いて差分だけ伝える
- 手段は固定しすぎず、解き方は相手の強みを活かす
- 詳細指定は、事故コストが高いところだけに絞る

## 典型ユースケース

- ペアプロの相手として使う（設計 -> 実装 -> レビューを同じ文脈で回す）
- 決定レビュー役として使う（重要判断の抜け漏れやリスクを継続確認する）
- 言語化チェック役として使う（説明文、方針文、PR文面の明確さを検証する）
- 影響範囲チェック役として使う（変更のたびに同じ観点で確認する）
- しりとりを継続して遊ぶ（同じ相手・同じ文脈で会話を続ける）

## 話しかけ方の目安

最初は短く投げて、必要になった分だけ追加する。

1. 最初に伝えること
   - 目的
   - 完了条件
2. 足りないときに足すこと
   - 触ってよい/避けたいファイル
   - 制約（互換性、期限、禁止事項）
3. 強く指定する場面
   - 手順固定、実装方式固定、出力形式固定
   - 理由があるときだけ使う

## 依頼の例

短文で十分な場合:

```text
目的: ○○を解決したい
完了条件: △△なら完了
```

追加情報が必要な場合:

```text
目的: ○○を解決したい
完了条件: △△なら完了
制約: ××は変更しない、互換性は維持
```

## 文脈を活かすコツ

- 同じエントリには「前回からの差分要求」だけ送る
- 会話が混線したら `/clear` して再開する
- エントリごとに責務を固定する（例: `review`, `impl`, `qa`）
- 並行送信時は、各エントリの問いを重複させない
- 重要な決定では「結論だけでなく理由も」渡して、次回レビューに繋げる

## 決定レビューの使い方

重要な判断をしたら、短く共有してチェックしてもらう。

```text
決めたこと: ○○にする
理由: △△
懸念: ××が残っている
確認してほしい点: 見落としや破綻がないか
```

## 言語化チェックの使い方

説明文のドラフトを渡して、誤解されやすい点を洗い出してもらう。

```text
目的: この説明がチーム内で誤読されないか確認したい
文面: （本文）
見てほしい点: 曖昧語、前提漏れ、読み手依存の表現
```

## 細かく指定するかの判断

- 細かい手順まで渡すほうがよいケース:
  - 事故コストが高い変更（本番影響、セキュリティ、破壊的移行）
  - 既存規約への厳密準拠が必要
- 任せるほうがよいケース:
  - 解法探索、設計案比較、改善提案
  - 同一PJ前提で、相手が必要情報を自力で取得できる

## エラー

| reason | 意味 | 対処 |
|--------|------|------|
| `:not_found` | エントリ名が存在しない | `entries.exs` で利用可能なエントリ名を再確認する |
| `:busy` | エントリが処理中 | 待ってリトライする |
| `"failed"` / `"interrupted"` | LLM 側でエラー | リトライする |
